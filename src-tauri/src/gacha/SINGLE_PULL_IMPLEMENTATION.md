# 单抽功能实现说明

## 概述

本文档描述了明日方舟抽卡系统中单抽功能的实现细节。

## 实现的功能

### 核心功能（需求 3.1-3.7）

1. **货币验证**（需求 3.1, 3.2）
   - 验证用户合成玉余额是否 >= 600
   - 余额不足时返回清晰的错误信息

2. **概率计算**（需求 3.3）
   - 调用 `calculate_gacha_rarity` 函数
   - 根据保底计数器计算稀有度

3. **干员选择**（需求 3.4）
   - 从对应稀有度的干员池中随机选择
   - 当前实现包含 40 个干员（每个稀有度 10 个）

4. **干员收藏**（需求 3.5）
   - 将获得的干员保存到数据库
   - 生成唯一的干员 ID

5. **保底计数器更新**（需求 3.6）
   - 获得 6★ 时重置为 0
   - 未获得 6★ 时加 1

6. **抽卡历史记录**（需求 3.7）
   - 保存完整的抽卡记录
   - 包含时间戳、类型、干员、消耗货币、保底计数器变化

## 函数接口

### `perform_single_gacha`

```rust
pub fn perform_single_gacha(conn: &Connection) -> Result<GachaResult, String>
```

**参数**:
- `conn`: SQLite 数据库连接

**返回值**:
- `Ok(GachaResult)`: 抽卡成功，包含获得的干员、更新后的保底计数器、消耗的货币
- `Err(String)`: 抽卡失败，包含错误信息

**错误情况**:
- 合成玉不足（< 600）
- 数据库操作失败

### `select_random_operator`

```rust
fn select_random_operator(rarity: u8) -> Operator
```

**参数**:
- `rarity`: 稀有度（3-6）

**返回值**:
- 随机选择的干员实例

2. `test_single_gacha_success` - 测试成功抽卡的完整流程
3. `test_pity_counter_update` - 测试保底计数器的正确更新
4. `test_operator_saved_to_database` - 测试干员保存到数据库
5. `test_gacha_history_saved` - 测试抽卡历史记录保存

所有测试均通过。

## 数据流程

```
1. 加载货币余额
2. 验证合成玉 >= 600
3. 加载保底计数器
4. 计算稀有度（调用概率引擎）
5. 从干员池随机选择干员
6. 更新保底计数器
7. 扣除 600 合成玉
8. 保存干员到数据库
9. 更新货币余额
10. 更新保底计数器
11. 保存抽卡历史记录
12. 返回抽卡结果
```
